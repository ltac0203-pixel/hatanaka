# API トークン運用

`/api/login` で発行する Sanctum 個人アクセストークンの取り扱いを記述する。

## 既定の動作

- `device_name` を指定して `/api/login` を呼ぶと、その `device_name` に紐づく Personal Access Token を発行する
- `abilities` は `subscription:read`, `subscription:write`, `card:read`, `card:write` のホワイトリストから明示要求する。未指定時のデフォルトは最小権限 `['subscription:read', 'card:read']`
- 寿命は次の通り:
  - 読み取り専用トークン (`*:read` のみ): **30 日**
  - 書き込み権限を含むトークン (`*:write` を含む): **7 日** (漏洩時の被害を抑えるため短縮)

## ローテーション (重要)

`/api/login` を成功するたびに、**同じ `device_name` の既存トークンは破棄される**。これによって以下を達成する:

- 漏洩した古いトークンが寿命まで併用されるリスクを排除
- 再ログインを実行することで失効を確定
- ユーザーに「再ログイン = 強制ローテーション」の操作モデルを提供

## モバイル/監視/Bot 等の長期稼働クライアントへの影響

長期間プロセスを継続するクライアント (モバイルアプリ、監視 daemon、Bot 等) では、以下のいずれかの **再認証ロジックを必ず実装**する必要がある:

1. **トークン期限の事前検出**
   - レスポンスの `expires_at` を保存し、期限の数日前から再ログインフローへ誘導
   - 書き込みトークンは 7 日寿命のため、週次の再認証 UI / 自動再ログインジョブを設計

2. **401 Unauthorized 受信時の再認証フロー**
   - リクエスト失敗時に 401 を検知 → 認証情報の再入力 (またはセキュアに保存した資格情報での再ログイン)
   - 失敗時はユーザー操作を要求する画面に遷移

3. **同一 `device_name` での同時ログインに注意**
   - 複数端末で同じ `device_name` を使うと、後発のログインが先発のトークンを破棄する
   - 端末ごとに固有な識別子 (例: `ios-iphone15-{uuid}`、`monitor-prod-1`) を `device_name` に使う

4. **abilities は最小権限を要求**
   - 監視・表示専用のクライアントは `abilities=["subscription:read","card:read"]` のみ要求
   - 書き込みが本当に必要な操作 (解約・カード追加) のときだけ `*:write` を要求し、寿命短縮の負担と被害局所化のトレードオフを意識

## CSRF / 認証経路

`/api/login` は **`throttle:api-login`** で保護されている。

- email + IP の複合鍵で 5 回/分
- IP 単位で 20 回/分

ブルートフォース・辞書攻撃に対する第一の防御線。クライアント側で再認証ループに陥ると即座にレート制限に引っかかるため、リトライには指数バックオフを実装すること。

## 旧トークン引き継ぎが必要な場面

「ローテーションせず古いトークンを使い続けたい」シナリオは設計上想定していない。継続利用には次のいずれかを検討:

- ユーザー側で意図的に再ログインを発火し、新トークンへ切り替える
- 別の `device_name` を使った 2 つ目のトークンを発行 (旧トークンは寿命まで生存)
