# Gitコミットガイドライン

English version: [commit-guidelines.md](./commit-guidelines.md)

このドキュメントでは、プロジェクトにおけるコミットの粒度とメッセージの規約を定義します。

## コミット粒度の原則

### 1コミット = 1つの意図（目的）

コミットは「1つの明確な目的」を持つ変更単位とします。

### サイズ感の目安

| 範囲    | 推奨度 | 備考                             |
| ------- | ------ | -------------------------------- |
| ~200行  | 理想   | レビューしやすい                 |
| ~400行  | 許容   | これ以上は分割を検討             |
| 400行超 | 要分割 | 例外: 自動生成ファイル、一括整形 |

## 粒度を決める実践ルール

### ルール1: 「戻せる単位」で切る

「このコミットだけrevertしたい」が自然にできる単位にする。

```
❌ UIの見た目調整とAPI仕様変更を同じコミットに混ぜる
⭕ UIの見た目調整とAPI仕様変更を別コミットにする
```

### ルール2: "機能"と"準備"を分ける

```
⭕ 良い例（分割されている）
1. ルーティング追加（画面の土台）
2. APIクライアント追加（呼び出しだけ）
3. 表示ロジック実装
4. テスト追加

❌ 悪い例（全部混ざっている）
1. 画面追加・API追加・ロジック・テストが全部1コミット
```

### ルール3: リファクタと挙動変更は混ぜない

- リファクタ（命名、整理、抽出）と仕様変更は別コミット
- 後から「どっちの変更で壊れた？」が追いやすくなる

### ルール4: フォーマット/リンタは"単独コミット"

Prettier / ESLint / pint などの**全体整形は他変更と混ぜない**。
→ 差分が埋もれてレビューが地獄になるため

```
⭕ style: pintでコードフォーマット  （単独コミット）
⭕ feat: ユーザー検索機能を追加     （別コミット）

❌ feat: ユーザー検索機能追加 + フォーマット修正
```

## コミットメッセージ規約

> **Note:** `public/build/` は `.gitignore` でGit管理対象外です。ビルドアセットのコミットは不要です。

### プレフィックス

| プレフィックス | 用途 |
| --- | --- |
| `feat:` | 新機能追加 |
| `fix:` | バグ修正 |
| `refactor:` | リファクタリング（機能変更なし） |
| `docs:` | ドキュメントのみの変更 |
| `style:` | フォーマット変更（コードの意味に影響しない） |
| `test:` | テストの追加・修正 |
| `chore:` | ビルド・補助ツールの変更 |

## 分割が推奨される例

変更が複数の目的を含む場合は、以下のように分割を推奨します：

```
⚠️ 変更内容が複数の目的を含んでいます。以下のように分割を推奨します：

コミット1: style: pintでコードフォーマット
  - backend/Controllers/*.php のフォーマット変更

コミット2: feat: ユーザー検索APIを追加
  - backend/Controllers/UserController.php の検索メソッド
  - src/api/user.ts のAPIクライアント

コミット3: feat: ユーザー検索UIを追加
  - src/pages/UserSearch.tsx
  - src/components/user/SearchForm.tsx
```

## コミットサイズの調整

### コミットが大きすぎる場合

- 複数の目的が混在している場合は、`git reset HEAD~` で戻して分割してください
- スコープが拡大した場合は、`git add -p` でハンク単位にステージし、別コミットに分けてください

### コミットが細かすぎる場合

- **typo修正が3連続など**の場合は、squashを検討してください
- `git rebase -i HEAD~N` でコミットをまとめることができます

## 注意事項

- ステージされていない変更は自動的にはコミットしません
- `.env` や認証情報を含むファイルはコミットしないでください
- 既存のコミットスタイル（日本語/英語）を継承します
- **コミットがデカすぎる**と感じたら、分割を検討してください
- **コミットが細かすぎる**（typo修正が3連続など）場合は、squashを検討してください
